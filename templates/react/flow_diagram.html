2 · Django Serves the SPA Shell
Portal Shell
GET /app/ → index.html
Django renders templates/react/index.html — a bare HTML page that loads the React JS bundle. React Router then takes over all /app/* navigation client-side.
spa_view() catch-all /app/<path>
Control Shell
GET /control-app/ → staff_index.html
Same pattern. Django serves templates/react/staff_index.html. React Router handles all /control-app/* sub-routes. No full-page reloads after initial load.
staff_spa_view() catch-all /control-app/<path>
React boots → calls /api/portal/ or /api/staff/
3 · Every API Call
Request pipeline — same for every fetch()
same-origin no CORS needed no JWT token automatic session cookie
  • 1 React calls fetch("/api/portal/dashboard/stats/", { credentials: "include" })
  • 2 Browser automatically attaches sessionid cookie (set at login, HttpOnly)
  • 3 For POST/PATCH/DELETE React also reads csrftoken cookie and sends it as X-CSRFToken header
  • 4 Django SessionAuthentication validates the session → sets request.user
  • 5 Permission class checks the user type (PortalSessionPermission or StaffSessionPermission)
  • 6 View executes → returns JSON → React re-renders
routed by Django URL conf

6 · Authentication — Option A (current) vs future
Option A · Now
Django Session
Same-origin session cookie. Login via existing /portal/login/ or Google OAuth. Zero extra dependencies. React just uses fetch(..., { credentials: "include" }).
sessionid cookie X-CSRFToken header HttpOnly
Option B · Future
JWT (djangorestframework-simplejwt)
When a mobile app or third-party client needs auth. Swap SessionAuthentication → JWTAuthentication in each view class. All 36 endpoints stay identical — only the auth mechanism changes.
Authorization: Bearer <token> access + refresh tokens
Option C · Vision
Microservices
Each domain (accounts, sales, loyalty) becomes an independent service. An API gateway validates JWTs and routes requests. The Django endpoints become the foundation each service is extracted from.
API gateway shared JWT issuer service mesh

Key Rules
React never computes points. The backend annotates pts on every Sale queryset using round(qty × product_ppv × points_per_unit). React receives a number and renders it. If the formula ever changes it changes in one place — fidpha/services.py:get_account_points_queryset().
The machine API (/api/v1/) is untouched. Pharmacy software submits sales via POST /api/v1/sales/ using Authorization: Token <raw>. This flow is completely separate from the React session flow. Both coexist without conflict.
Django templates stay alive during the migration. /portal/ and /control/ routes remain fully functional. React replaces pages one by one — there is no big-bang cutover.
